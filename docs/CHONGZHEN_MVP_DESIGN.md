# 《历史模拟器：崇祯》设计文档（严格晚明制度版）

## 1. 项目定位

《历史模拟器：崇祯》是一款以晚明制度失灵为核心玩法的 AI 原生历史策略游戏。

玩家扮演 1627 年登基的崇祯帝，面对的是一个并非“命令一出、全国照办”的国家机器，而是一个法理上皇权独尊、现实中权力链条层层扭曲、行政过程高度依赖人事关系、财政承压、地方执行和政治摩擦的晚明帝国。

本作的核心体验不是“选择正确答案”，而是：

- 你可以下诏，但诏令未必能落地
- 你可以整顿，但整顿会制造新的阻力
- 你可以催饷守辽，但会压垮腹地财政与民心
- 你可以借内廷破局，但会污染外朝秩序
- 你可以压制党争，但可能只是把公开冲突改成隐蔽离心

因此，本作的后端规则引擎，不应按“现代行政分工”设计，而应按“晚明真实政治运行方式”设计。

---

## 2. 设计总纲

### 2.1 核心历史口径

本作模拟的不是一个机构分工清晰、行政链条稳定的现代国家，而是一个在皇权独尊之下、名义制度与现实运作长期错位的晚明帝国。

皇帝是唯一法理上的最高决策者，但政令不会因为圣旨下达而自动落实。所有政策都必须穿过外朝、内廷、部院、科道、督抚、地方官与社会中介网络，才能转化为真实效果。

晚明政治在游戏中的基本判断如下：

1. 皇帝拥有最高裁决权，但皇帝个人性格、精力、疑心与信任结构会改变整个国家机器的运作方式。
2. 内阁拥有票拟与政务协调影响力，但并不是法定宰相机关，其权力来自皇帝授权、政治惯例与阁臣威望。
3. 司礼监掌批红与宫中文书枢纽，其强弱并非制度常态，而是随皇帝对内廷的依赖程度而变化。
4. 六部是中央执行骨架，但执行过程长期受制于财力不足、积案迟滞、胥吏侵蚀、地方抗拒与言官封驳。
5. 科道言官既是监察工具，也是政治斗争力量，能够纠错，也会放大治理摩擦。
6. 晚明军政运行已不能简单理解为“兵部 vs 五军都督府”二元结构，真正关键的是兵部、督抚、边镇将领、军饷与战区协调体系。
7. 厂卫是皇帝绕开常规外朝法统的非常手段，能短期提效，也会长期破坏合作与合法性。

### 2.2 玩家体验目标

玩家应该体验到：

- 皇命可下，执行有价
- 财政是所有政策的硬边界
- 军事不是单纯堆兵，而是饷、将、镇、调度、猜忌的综合结果
- 官僚体系不是静态资源，而是会顺从、拖延、曲解、抵制、互相攻讦的活结构
- 每次决策都可能在几个月后以另一种形式反噬

### 2.3 规则总原则

本作规则引擎遵循 6 条原则：

1. 皇权至上，但皇权不等于万能
2. 政令必须经过制度链条
3. 执行过程一定存在损耗
4. 机构之间既合作也博弈
5. 地方反馈决定政策最终效果
6. 系统只结算代价，不提供道德判定

## 3. 游戏时间与范围

### 3.1 时间范围

- 起点：1627 年 10 月
- 终点：1630 年 12 月
- 总计：39 回合
- 每回合：1 个月

之所以将首版时段锁定在这三年多，是因为这一阶段足以覆盖：

- 天启末年权力遗产的延续
- 魏忠贤体系余波
- 崇祯初政的人事震荡
- 陕西灾荒与流民问题恶化
- 辽东边防持续紧张
- 晚明财政与党争结构性失稳

5. 山西
6. 河南
7. 山东

9. 江南

## 4. 晚明制度结构建模

这一部分是后端规则引擎的根基。

## 4.1 皇帝

皇帝是唯一法理上的最高决策者，但并不是“直接执行者”。

皇帝在系统中的作用是：

- 决定议题优先级
- 决定是否亲裁
- 决定是否重用外朝
- 决定是否依赖内廷
- 决定是否以非常手段压制阻力
- 决定是否牺牲某一体系换取短期效率

### 皇帝核心属性

- `prestige` 威望
- `legitimacy` 正统性
- `diligence` 勤政度
- `stress` 压力
- `trust_civil` 对文官信任
- `trust_eunuch` 对内廷信任
- `trust_military` 对边将信任
- `suspicion` 猜忌度

### 设计要求

皇帝不直接提供资源或数值增益，而是改变“国家机器怎么运行”。

例如：

- `trust_civil` 高：更依赖内阁与部院，流程更规范，但速度较慢
- `trust_eunuch` 高：更依赖司礼监与厂卫，短期推进更快，但外朝敌意上升
- `suspicion` 高：更容易频繁换将、否决票拟、引发军政失序

### 中旨机制

严格晚明语境下，必须把 `中旨` 单列出来。

中旨不是普通诏令，而是皇帝自内廷直接发出的旨意，可能绕开部分正常外朝程序，或在程序未完备时强行定夺。

在游戏中，中旨应被建模为一种特殊政策通道，而不是单纯 flavor 文本。

#### 中旨的典型特征

- 速度快
- 可绕开部分票拟与外朝协调
- 对内廷依赖高
- 合法性争议大
- 更容易引发科道、部院与士林反弹

#### 中旨的系统效果

正面：

- 降低 `delay_days`
- 在紧急军事、缉捕、内廷人事与特殊处置时提升推进力

负面：

- 增加 `political_backlash`
- 提高 `censor_review` 阻力
- 提高“名分受损”“外朝离心”“程序争议”类事件概率

#### 中旨的适用范围

游戏中不应允许所有事务都走中旨。

更合理的设计是：

- 军事情急
- 厂卫侦缉
- 内廷相关人事
- 对个别臣僚的特殊处分

而涉及全国税制、长期财政、大规模制度调整时，中旨应更容易引发严重制度反噬。

## 4.2 内阁

内阁不能被写成正式宰相机关。

在严格晚明语境下，内阁是：

- 皇帝身边的票拟与政务协调机构
- 通过处理章奏、起草意见、协调部务来影响国家运行
- 其权力大小取决于皇帝授权、阁臣资望、外朝合作与时局需要

### 内阁的历史定位

- 内阁不是法定“相府”
- 首辅不是法定丞相
- 内阁权力的本质，是在无相体制下因事务密集与政治惯例而形成的事实中枢影响力

### 内阁在游戏中的功能

- `draft_memorial_response` 票拟
- `prioritize_memorials` 处理章奏优先级
- `coordinate_six_ministries` 协调六部
- `moderate_edict` 规范或缓和诏令表述
- `delay_or_soften` 拖延或消解皇帝原意

### 内阁核心属性

- `competence`
- `cohesion`
- `civil_support`
- `faction_bias`
- `loyalty_emperor`
- `resistance_to_eunuch`

### 系统效果

正面：

- 提高政策结构化质量
- 降低粗暴命令的偏差
- 增加部院配合度

负面：

- 可能拖延
- 可能曲解
- 可能借规范程序削弱皇帝原意

---

## 4.3 司礼监

司礼监不能被设计成恒定存在、稳定对等于内阁的“第二政府”。

在严格历史口径下，司礼监的权力强弱是动态变量，不是常量。

### 司礼监的历史定位

- 掌批红
- 掌宫中文书通道
- 能否压倒外朝，取决于皇帝是否怠政、是否倚重内廷、是否将实际裁断交给秉笔太监

### 崇祯初年的建模要求

崇祯初年不应直接设定为“司礼监永远强势”，而应设定为：

- 内廷仍保留制度惯性和影响网络
- 但皇帝正在重新整肃、压缩、重组其权力空间
- 它既是可利用工具，也是潜在政治污染源

### 司礼监在游戏中的功能

- `red_ink_approval` 批红
- `accelerate_execution` 加速政令下达
- `bypass_some_civil_steps` 绕开部分外朝摩擦
- `link_eastern_depot` 联动厂卫

### 核心属性

- `influence`
- `dependency_of_emperor`
- `efficiency`
- `corruption`
- `hostility_from_civil`

### 系统效果

正面：

- 缩短流程时延
- 提高强制推进能力

负面：

- 强化阉党残余或内廷势力
- 激化文官与科道敌意
- 增加暗箱操作与寻租概率

---

## 4.4 六部

六部不是现代“部长制内阁”。

严格来说，六部是中央成命的主要承接者，是政务执行骨架，但其执行效果受制度、财政、人员、地方和舆论多重限制。

### 六部的统一设计原则

六部每一部都需要同时具备：

- 法理职权
- 实际承载能力
- 积案压力
- 部内胥吏侵蚀
- 派系偏向
- 与科道、内阁、地方的摩擦值

### 吏部

主管文官任免、考核、升降。

重点不是“任命谁”，而是：

- 用人是否得其人
- 是否破坏既有官场平衡
- 是否引发派系性反扑

关键字段：

- `appointment_fairness`
- `cadre_efficiency`
- `patronage_level`

### 户部

主管财政、赋税、军需、漕运、银粮调配。

晚明游戏逻辑里，户部必须是核心约束机构。

原因：

- 绝大多数政策最终都要花钱
- 军事能否维持，取决于户部能否持续供饷
- 救灾能否见效，取决于户部是否真能拨到地方

关键字段：

- `silver_balance`
- `grain_logistics`
- `tax_efficiency`
- `fiscal_flexibility`

### 礼部

主管典礼、学校、科举、名分与藩属礼制。

其作用不在直接执行财政或军事政策，而在：

- 维护正统性
- 影响士林舆论
- 决定政治行为是否“名义上站得住”

### 兵部

主管军政调度、武官任命、兵籍与战略统筹。

但必须强调：

- 兵部不等于拥有实兵
- 兵部的调度能力受边镇、督抚、饷源、前线将领关系制约

### 刑部

主管司法行政与重要案件审理。

其历史角色不是简单“抓人”，而是：

- 为整顿和镇压提供法理包装
- 与都察院、大理寺共同构成法统一面
- 在高压政治中可能被拉去为政治行动善后

### 工部

主管工程、水利、屯田、军工、修缮。

其在晚明危局中的重要性是：

- 救灾长期恢复
- 水利修复
- 城防与军械生产
- 地方生产恢复

---

## 4.5 科道言官：都察院与六科

科道必须被建模为“制度摩擦层”，而不是单纯的纠错装置。

### 都察院

历史作用：

- 监察百官
- 巡按地方
- 弹劾失职与贪污

### 六科给事中

历史作用：

- 对应监察六部
- 审核政令合理性
- 以封驳、封还、质疑方式制造程序阻力

### 游戏中的严格定位

科道言官既是皇权纠察百司的工具，也是士大夫政治表达与派系斗争的重要载体。

它们会同时带来：

正面：

- 揭发贪墨
- 暴露失政
- 修正过激命令

负面：

- 放大程序拖延
- 放大名义合法性冲突
- 借公议行党争

### 关键机制

对部分政令，科道系统可产出：

- `allow`
- `delay`
- `revise`
- `impeach`
- `reject`

---

## 4.6 军政体系

这一部分必须按晚明实际改写，不能再简单写成“五军都督府与兵部二元分权”。

### 历史修正口径

祖制上，明朝确有兵部与五军都督府分掌军政、军籍、调度的制度设计。

但到晚明，尤其崇祯时期，真正影响战争结果的核心已转为：

- 兵部
- 经略 / 督师
- 总督 / 巡抚
- 边镇将领
- 军饷与转运系统
- 家丁与边军实际战力网络

五军都督府在游戏中不应再被设计为前线军权核心，而应被弱化为：

- 旧制军籍与名义祖制残影
- 部分军制阻力参数
- 编制与法理层面存在感

### 游戏中的军政链条

严格晚明版应写成：

1. 皇帝裁决
2. 兵部下达调度与任命
3. 督师 / 经略 / 总督 / 巡抚进行战区统筹
4. 边镇总兵、副将、参将落地执行
5. 户部与地方供饷
6. 前线军纪、士气、粮饷和将帅关系共同决定结果

### 军事系统核心体验

- 有令未必有兵
- 有兵未必有饷
- 有饷未必能及时送到
- 有将未必可信
- 换将可能更糟
- 皇帝猜忌会直接损伤边防稳定

---

## 4.7 厂卫

厂卫不能只被写成“特务组织”。

严格讲，它们是皇帝绕开常规外朝法统的非常治理工具。

### 锦衣卫

- 皇帝亲军与诏狱系统
- 兼具侦缉、逮捕、仪卫与情报功能

### 东厂

- 由内廷外伸的监视网络
- 能监督百官，也能监督锦衣卫
- 与司礼监联动时威力更强

### 游戏中的历史定位

厂卫的危险不只在“恐惧值上升”，更在于：

- 破坏常规司法与外朝秩序
- 让士人、台谏与部院失去安全感
- 造成诬告、冤狱与行政寒蝉

### 系统效果

正面：

- 快速侦缉
- 高压施压
- 提升短期控制力

负面：

- 降低文官合作
- 降低法统稳定
- 提升政治反噬
- 伤害皇帝长期名望

## 5. 游戏核心循环

每回合为 1 个月。

### 5.1 回合流程

1. 月初局势简报
2. 玩家查看全国地图与图层
3. 玩家发起廷议或召对
4. 玩家拟写诏书
5. 系统解析诏书
6. 诏书进入“朝廷流程层”
7. 朝廷流程结算后进入“地区执行层”
8. 更新世界状态
9. 触发动态事件
10. 生成朝堂奏报、民间反应与派系反馈
11. 进入下一回合

### 5.2 关键体验要求

诏书不能直接改数值，必须体现：

- 谁经手了这道诏令
- 哪一层拖了它
- 哪一层歪曲了它
- 哪些成本在途中流失
- 哪些政治力量因此被激怒

## 6. 诏书与政策系统

### 6.1 诏书输入

玩家使用自然语言拟诏。

系统不要求玩家使用标准格式，但后端必须解析为结构化政策。

### 6.2 政策类型

首版保留 10 类：

1. `disaster_relief` 赈灾
2. `grain_transfer` 调粮
3. `tax_adjustment` 调税
4. `military_supply` 军饷军资
5. `recruitment` 招募与练兵
6. `anti_corruption` 整饬吏治
7. `faction_purge` 打击派系
8. `official_appointment` 人事任命
9. `infrastructure` 工程水利
10. `intelligence_crackdown` 厂卫侦缉

### 6.3 政策结构化字段

- `policy_type`
- `target_regions`
- `target_institutions`
- `target_faction`
- `budget_silver`
- `grain_amount`
- `strictness`
- `urgency`
- `duration_months`
- `political_cover`

### 6.4 双层解析机制

第一层：LLM 理解自然语言意图

第二层：规则纠偏

规则纠偏负责：

- 补默认值
- 校验非法组合
- 修正缺失字段
- 给出 deterministic fallback

### 6.5 政令来源类型

为符合晚明制度，政策结构化后还应区分 `order_channel`。

建议至少区分：

- `regular_edict` 常规诏令
- `cabinet_memorialized` 经票拟整理的成命
- `zhongzhi` 中旨
- `military_dispatch` 军事情急调度

这会直接影响后续的流程阻力与政治反噬。

## 7. 朝廷流程层

这是本作最关键的规则层。

### 7.1 为什么必须存在

如果没有这一层，游戏会退化成“玩家写一句话，数值立刻变化”的资源管理游戏，而不是晚明制度模拟游戏。

### 7.2 基本思想

每条诏书形成后，不直接生效，而是先进入“法理链条”和“实际链条”。

### 7.3 法理链条

法理上，一道政令可能经过：

1. 皇帝裁断
2. 内阁票拟
3. 司礼监批红
4. 通政与文移流转
5. 六部承办
6. 科道监督
7. 地方官执行

### 7.4 实际链条

现实中，一道政令真正能否落实，还要看：

1. 皇帝是否持续关注
2. 内阁是否愿意认真整理
3. 司礼监是否加速或插手
4. 通政、誊黄、传抄与文移是否顺畅
5. 部院是否有钱、有力、有人
6. 科道是否封驳、弹劾、借题发挥
7. 地方督抚是否配合
8. 胥吏、豪强、运输与灾情是否吞噬政策效果

### 7.5 朝廷流程节点

首版建议统一为五段：

1. `cabinet_review`
2. `eunuch_review`
3. `transmission_review`
4. `ministry_processing`
5. `censor_review`
6. `local_execution`

### 7.6 每一节点可能结果

- `pass`
- `delay`
- `distort`
- `resist`
- `over_enforce`

特别说明：

- `distort` 指政令被改样、缩水或曲解
- `over_enforce` 指命令被执行过头，例如催饷演变为地方横征暴敛

### 7.7 中旨在流程层中的特殊路径

若 `order_channel = zhongzhi`，则流程不应完全等同于常规诏令。

建议首版按以下方式处理：

- 可跳过或弱化 `cabinet_review`
- 强化 `eunuch_review`
- 直接进入 `transmission_review` 或 `ministry_processing`
- 在 `censor_review` 环节追加额外阻力

也就是说，中旨的本质不是“无阻力”，而是“前段加速、后段反噬”。

## 8. 执行率与损耗模型

### 8.1 执行率不是随机值

执行率必须由多因子决定：

- 皇帝威望
- 皇帝信任结构
- 牵头机构效率
- 牵头机构忠诚
- 牵头机构腐败
- 派系敌意
- 地方官承载能力
- 区域交通与距离
- 灾情严重度
- 政策复杂度

### 8.2 推荐公式

```text
execution_rate =
  base
  + emperor_prestige * a
  + lead_institution_efficiency * b
  + local_capacity * c
  - corruption * d
  - faction_resistance * e
  - transport_penalty * f
  - disaster_penalty * g
```

最终限制在 `5 ~ 95`。

### 8.3 必须计算的三类损耗

1. `delay_days` 时间损耗
2. `corruption_loss` 腐败损耗
3. `transport_loss` 运输损耗

### 8.4 必须计算的三类反噬

1. `political_backlash`
2. `bureaucratic_chill`
3. `public_burden_shift`

例如：

- 用东厂压办，`delay_days` 下降，但 `political_backlash` 上升
- 加税保辽，边防略稳，但 `public_burden_shift` 激增
- 严查贪墨，短期效率上升，但 `bureaucratic_chill` 上升

## 14. 大臣廷议系统

大臣不能只是说几句 flavor 文本，而应作为“结构化建议代理”。

### 14.1 大臣需要具备

- 身份
- 职位
- 所属机构
- 派系归属
- 性格
- 政策偏好
- 能力
- 廉洁度
- 野心
- 忠诚

### 14.1.1 廷议与召对的区分

严格制度上，游戏里不应把所有“问策”都叫廷议。

建议分成两类：

#### 召对

- 皇帝单独召见 1-2 名大臣
- 偏私密
- 偏快速
- 更适合形成中旨、密令、临时判断

#### 廷议

- 高规格公开议政
- 参与者是中央最高统治阶层
- 更慢，但信息更全面
- 更容易形成制度性共识，也更容易暴露公开分裂

也就是说，前端“召集廷议”按钮在最终文案上可以保留，但后端应区分：

- `audience_mode = private_consultation`
- `audience_mode = grand_court_conference`

### 14.1.2 廷议参与者

严格按晚明制度，廷议规格极高，参加者应按“科道九卿”与相关近臣建模。

首版建议将参与者分为 6 类：

1. 六部尚书、侍郎
2. 都察院左右都御史
3. 大理寺卿
4. 通政司使
5. 詹事府、翰林院高级官员
6. 六科给事中与十三道监察御史

如果要压缩实现，可以先做“席位组”而不是逐人逐官全量建模。

#### A. 六部堂官

- 吏部尚书、侍郎
- 户部尚书、侍郎
- 礼部尚书、侍郎
- 兵部尚书、侍郎
- 刑部尚书、侍郎
- 工部尚书、侍郎

作用：

- 提供各部执行视角
- 暴露资源、流程、法理和技术层面的可行性

#### B. 都察院堂官

- 左都御史
- 右都御史

作用：

- 从监察角度指出失政、贪墨、地方执行偏差

#### C. 大理寺卿

作用：

- 从法理与审案角度评估整顿、缉捕、诛谴是否失当

#### D. 通政司使

作用：

- 代表章奏、题本、文移流转视角
- 让“信息到皇帝耳中”和“皇命传到地方”成为一个可结算环节

#### E. 詹事府、翰林院高级官员

作用：

- 提供诏令文义、名分、士林观感与政治修辞层面的影响

#### F. 六科给事中与十三道监察御史

作用：

- 代表言路压力
- 代表封驳可能
- 代表朝堂公议与公开反对声

### 14.1.3 廷议不是所有人都逐个发言

为控制实现复杂度，游戏里不应强制逐个生成几十名官员发言。

更合理的做法是：

- 把廷议参与者抽象成若干“席位组”
- 每组生成 1 条立场摘要
- 再选 2-4 名关键人物输出具名发言

这样既保留历史结构，也不让交互失控。

### 14.2 廷议输出应包括

- `stance`
- `recommended_policies`
- `warning_tags`
- `institutional_path`
- `likely_backlash`
- `content`

如果是高规格廷议，建议额外返回：

- `attendees`
- `seat_group_opinions`
- `consensus_level`
- `public_disagreement_level`
- `recommended_order_channel`

### 14.3 廷议的真实作用

- 不是告诉玩家“最佳答案”
- 而是让玩家看见不同政治路径的代价

### 14.4 廷议对后续诏书的影响

廷议结果不应只影响文案，而应真实进入规则层：

- 提高或降低后续政策的 `legitimacy_bonus`
- 改变 `cabinet_review` 阻力
- 改变 `censor_review` 阻力
- 改变某些派系 `hostility`
- 影响系统建议玩家更适合走 `regular_edict` 还是 `zhongzhi`

## 19. 文档结论

这份设计文档的核心修正只有一句话：

**本作模拟的不是“皇帝直接操纵数值”，而是“皇帝试图驱动一台已经严重扭曲、失衡、互相掣肘的晚明国家机器”。**

因此：

- 皇帝是最高裁断者
- 内阁是票拟与协调中枢，不是正式相府
- 司礼监是可变强弱的内廷枢纽，不是恒定第二政府
- 六部是执行骨架，但远非万能
- 科道是监察与摩擦并存的制度层
- 军政核心在兵部、督抚、边将与军饷，而非抽象旧制
- 厂卫是非常手段，不是常规高效行政替代品

只有在这个前提下，《历史模拟器：崇祯》才真正会变成一个有历史质感、有制度逻辑、有 AI 原生推演价值的游戏。

## 20. AI 原生 Agent 与 MCP 架构设计

在《历史模拟器：崇祯》中，为了让朝堂真正“活起来”（如官员因局势变化自主投靠阉党、互相弹劾），本作采用了结合 **MCP（Model Context Protocol）理念** 与 **休眠-唤醒机制（Actor Model）** 的底层架构。

### 20.1 MCP 理念在游戏中的映射

MCP 的核心在于将“大模型推理”与“外部工具/环境”解耦。在本作中：

- **Game MCP Server（游戏世界引擎）**：
  引擎掌握着所有底层数据（国库、官员名册、灾情等），并向 Agent 暴露一组严格受控的“权力工具”（Tools）。例如：
  - `change_faction(minister_id, target_faction)`：改换门庭
  - `impeach_colleague(attacker_id, target_id, reason)`：上疏弹劾
  - `bribe_eunuch(minister_id, amount)`：打点厂卫
- **Agent Client（官员大脑）**：
  大语言模型本身不存储状态，仅负责接收当前视角的上下文（Context），进行心理博弈与推演，最终输出一个**结构化的 Tool Call**（函数调用），交由 Game Server 执行。

### 20.2 避免“全员常驻”的休眠-唤醒模型

本作**绝不会为 100+ 位官员保持 100 个常驻的并发 Agent**。这种做法在算力成本与延迟上是灾难性的。本作采用**事件驱动（Event-Driven）的按需唤醒机制**：

1. **筛选活跃分子**：每回合（每月）交替时，引擎先通过简单的代码逻辑（轻量级计算）扫描全体官员，找出“触发唤醒阈值”的角色（如忠诚度低于30、刚被廷杖、所在派系领袖被杀等）。
2. **组装局部视角的 Prompt**：仅为这几位被唤醒的官员构建当前局势的 Context，并提供可用的 MCP Tools。
3. **工具调用与状态固化**：被唤醒的 Agent 思考后调用工具（如“我要投靠阉党”），引擎执行修改，随后该 Agent 再次进入休眠。

通过此设计，90% 的普通官员平时处于极低能耗的“待机”状态，只有处于风口浪尖的官员才会消耗大模型算力，实现涌现性与成本的完美平衡。

### 20.3 如何解决 Context（上下文）规模爆炸的问题？

在复杂的历史模拟中，如果每次唤醒 Agent 都将全国地图、100名官员的履历、过去十年的邸报全部塞入 Prompt，会迅速撑爆大模型的 Context Window（上下文窗口），并导致幻觉和高昂成本。本作采用以下三种手段解决 Context 膨胀：

1. **图数据库（Neo4j）按需裁剪**：
   依靠后端构建的实体关系图谱。当唤醒“左良玉”时，系统只在图数据库中查询 `(左良玉)-[关系]-(周边武将/朝廷派系)` 的局部子图。Agent 只能看到自己能接触到的人脉和情报，而非上帝视角。
2. **动态摘要（Memory Summarization）**：
   官员的长期记忆不会以流水账形式堆积。后台会有轻量级的任务定期对过去的事件进行“记忆浓缩”。例如将“1月被罚俸、2月被弹劾、3月被降级”浓缩为一条核心设定：“你近期遭到了严重的政治打压，心怀怨恨”。
3. **宏观局势抽象化**：
   向 Agent 描述当前朝堂局势时，不传递所有官员的完整阵营分布，而是传递结构化的“权力大盘指数”（如：本月阉党权势滔天（影响力80），江南清流集团遭到清洗）。

这种“局部感知 + 结构化记忆 + 函数调用”的组合，是大明朝堂能够以极高复杂度且极低成本自发运转的基石。
