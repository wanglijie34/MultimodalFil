# 《历史模拟器：崇祯》核心设计文档（Master Design）

## 1. 项目定位与终极愿景

《历史模拟器：崇祯》是一款以晚明制度失灵为核心玩法的 **AI 原生历史策略大作**。
玩家扮演 1627 年登基的崇祯帝（朱由检），接手一个边患不断、阉党横行、财政崩溃、民心动荡的危局。游戏开局即面临陕西大旱、辽东后金威胁、国库空虚、党争激烈以及小冰河期天灾等多重危机，堪称“地狱开局”。

本游戏的核心设计理念是：**本作模拟的不是“皇帝直接操纵数值”，而是“皇帝试图驱动一台已经严重扭曲、失衡、互相掣肘的晚明国家机器”。**

本作的最大特色在于**基于大语言模型（LLM）驱动的无脚本动态推演系统**：
- **没有固定事件树**：所有历史人物均由独立的 AI 智能体（Agent）驱动，拥有各自的性格、立场与行为逻辑。
- **自然语言交互**：玩家用白话或半文言拟写诏书，与群臣廷议，AI 实时解析意图并推演连锁反应。
- **真正的蝴蝶效应**：整个明末的社会、经济、军事系统构成一个动态演化的“世界推演模型”，遵循“只算代价，不分对错”的物理法则，实现“千人千局”。

## 2. 核心游戏循环 (Core Loop)

游戏采用回合制策略玩法，每回合（1个月）涵盖完整的**空间、时间、金钱与政治威望的四维博弈**。

### 融合历史真实的一回合完整流程设计：

**📥 阶段一：月初局势简报与“拆盲盒”（第 1 - 5 天）**
- **世界状态更新**：小冰河天灾（每月修正）、各地在途军队与粮草的动态结算。
- **递交邸报与奏折**：案头堆起本月的臣工上疏。
- **派系干扰触发（信息迷雾）**：系统检测通政司是否被某派系把控（如开局的阉党）。若被把控，灾情图层和文字奏折将强制开启“粉饰太平”或“构陷异己”滤镜。玩家只能看到模糊或被篡改的数据，真正的危机正在暗处潜伏。

**👥 阶段二：廷议或召对（信息交锋与危机爆发）（第 6 - 15 天）**
- **发起交互**：
  - **选项 A（廷议）**：召集科道九卿大朝会。群臣在朝堂上为了各种议题疯狂扯皮、互相驳斥。
  - **选项 B（召对）**：私密召见个别重臣。大臣在私密状态下可能卸下政治防备，向皇帝吐露真言。
- **提炼决策核心**：通过大臣们的争吵和真言，玩家交叉比对情报，看清本月最致命的危机。

**✍️ 阶段三：下明诏与内阁票拟草案确认（第 16 - 20 天）**
- **自然语言拟诏**：玩家在宣纸文本框中输入自己的圣旨意图。
- **大模型解析与“二次确认”**：大模型实时读取意图，并在屏幕上弹出【内阁票拟草案】UI，将指令结构化（政令渠道、预计耗银、在途周期等）。玩家确认无误或微调后，点击“盖印”正式下发。避免 AI 识别偏差导致的毁灭性后果。

**🌀 阶段四：朝廷流程层穿透（The Bureaucratic Run）（第 21 - 30 天）**
- **自动化结算**：系统模拟圣旨在中央官僚体系中的流转。
- **流程摩擦**：圣旨路过内阁、六科、六部时，可能遭遇“磨合拖延”、“特权封驳”、“截留亏空”等情况。最终效果受玩家威望、派系敌意共同折算。

**📦 阶段五：月末结算与圣旨物流日志（第 31 天）**
- **生成追踪月报（Audit Trail）**：界面弹出清晰的【圣旨追踪日志】。玩家可明确看到钱在哪一步被谁贪了，政令在哪一层被谁卡了（如：“户部尚书以太仓空虚削减五万两”，“给事中上疏封驳延误 15 天”）。
- **危机升级与崩盘判定**：根据结算后的前线状态（如断饷导致哗变）更新危机。本回合结束，进入下一轮拉扯。

## 3. 可视化与地图视图

UI 侧重于通过地图来呈现复杂的国家数据，包含以下可切换图层（Overlay）：
1. **标准视图**：展示省份边界、主要城市与九边重镇。
2. **灾情视图**：以赤色热力图显示大旱、蝗灾、洪涝的严重程度。
3. **民心/动乱视图**：显示各地流民聚集地、李自成/张献忠等流寇的活动轨迹与蔓延趋势。
4. **城防/军压视图**：显示敌军（后金、流寇）的兵锋压力与己方卫所的防御强度。
5. **税收视图**：显示各省的抗税情况、宗藩兼并程度与库银充盈度。

## 4. 晚明制度结构建模

这一部分是后端规则引擎的根基。

### 4.1 皇帝（最高决策者）
皇帝不直接提供数值增益，而是决定“国家机器怎么运行”（议题优先级、亲裁与否、倚重内廷还是外朝）。
**皇帝威望（Prestige）的终极乘区效应**：
- **威望 > 80（雷霆万钧）**：执行系数 = 1.2，延迟惩罚 = 0.8。流程加速，前线震慑。
- **40 - 80（流于形式）**：执行系数 = 1.0，延迟惩罚 = 1.0。正常损耗与流程摩擦。
- **威望 < 40（政令不出乾清宫）**：执行系数 = 0.4，延迟惩罚 = 1.8。文官开启“合法敷衍”，顺从度断崖式下跌，政令被无限期拖延、篡改和层层剥皮。
- **中旨机制**：绕开外朝程序的特权。速度快，但高合法性争议，易引发严重制度反噬。

### 4.2 内阁（票拟与协调中枢）
内阁不是法定丞相。其本质是因事务密集形成的事实中枢。负责票拟、处理章奏、协调六部。能提高政策质量，但也可能拖延、曲解、消解皇帝原意。

### 4.3 司礼监（内廷枢纽）
掌批红与宫中文书。权力强弱动态变化，取决于皇帝对内廷的依赖。能加速执行、绕开外朝，但会激化文官敌意、增加寻租腐败概率。

### 4.4 六部（政务执行骨架）
- **吏部**：用人与考核，极易引发派系反扑。
- **户部**：**核心约束机构**。主管钱粮，绝大多数政策需户部供饷。
- **礼部**：维护正统性，影响士林舆论。
- **兵部**：军政调度，受饷源和前线将领牵制，且兵部不等于拥有实兵。
- **刑部**：司法与善后包装。
- **工部**：水利、军工、长期恢复。

### 4.5 科道言官（都察院与六科）
“制度摩擦层”。既揭发贪墨、修正过激命令，又放大程序拖延，借公议行党争（封驳、弹劾）。

### 4.6 军政体系
不再是简单的“五军都督府”，而是：皇帝裁决 -> 兵部调度 -> 督抚统筹 -> 边镇执行 -> 户部供饷。
核心体验：有令未必有兵，有兵未必有饷，有饷未必送到，有将未必可信。

### 4.7 厂卫（锦衣卫与东厂）
皇帝绕开常规法统的非常治理工具。能快速侦缉、高压施压，但会破坏常规秩序、造成行政寒蝉，带来严重的政治反噬。

## 5. 情报与派系系统

### 5.1 派系系统
派系服务于“制度错位下的政治摩擦”。包括：东林/清流、阉党/内廷、边军集团、财政官僚、地方士绅。影响票拟倾向、部院配合、科道弹劾和舆情。

### 5.2 奏折的“欺骗性”与信息去伪存真
朝堂被某派系把控时（如阉党），递上来的奏折是高度污染的“政治迷雾”。
- **情报过滤**：通政司会粉饰太平，掩盖真实灾情。
- **明枪暗箭**：言官弹劾往往是为了清除异己，而非吏治清明。
- **破局解法**：玩家不能偏听偏信，必须通过【厂卫侦缉】或【密召对】交叉比对数据，获取真实的法理证据。

---

## 6. 诏书与政策系统

### 6.1 诏书输入与解析
玩家使用自然语言拟诏。大模型不仅理解意图，还会经过**规则纠偏**（补默认值、校验组合），最终在 UI 提供**内阁拟定（二次确认）**。

### 6.2 结构化字段与损耗模型
政策包含目标区域、机构、预算、严厉度、紧急度等。
**三大损耗**：时间损耗（delay_days）、腐败损耗（corruption_loss）、运输损耗（transport_loss）。
**三大反噬**：政治反噬、官僚寒蝉效应、民间负担转移。

---

## 7. 大臣廷议系统

大臣作为“结构化建议代理”，拥有职位、性格、派系和野心。
- **召对**：私密召见 1-2 名大臣，适合探听真言与下达密令。
- **廷议**：高规格大朝会（六部、都察院、大理寺、六科等代表出席）。
为控制成本，系统将参与者抽象为“席位组”，仅选 2-4 名关键人物输出具名发言。廷议结果直接影响后续政策的法统加成与流程阻力。

---

## 8. 成长与破局系统

在“地狱开局”中，玩家需要从顶层改变大明国运：
- **国策规划 (National Policy)**：倾斜资源解锁核心国策，改变底层物理法则（如【厂卫治国】vs【众正盈朝】，【辽饷加派】vs【摊丁入亩】）。
- **科技树 (Tech Tree)**：引入徐光启、宋应星等人，推动技术转型（红夷大炮、抗旱番薯等）。

## 9. 胜负与结局判定

**崩盘条件**（满足其一即败）：
京师崩坏、威望归零、国库粮储双断、叛乱极高、辽东崩坏、行政瘫痪。

**阶段性评级**（1630年底结算）：
根据王朝维持、西北/辽东局势、财政党争等维度评估。成功不是中兴，而是“在坏局里延缓崩解”。

## 10. AI 原生 Agent 与 MCP 架构设计

为了让朝堂“活起来”且不致服务器破产，本作采用结合 **MCP（Model Context Protocol）** 与 **休眠-唤醒机制（Actor Model）** 的架构。

### 10.1 休眠-唤醒模型 (Event-Driven)
**绝不全员常驻**。90%官员平时待机，回合交替时仅扫描唤醒风口浪尖上的官员。为他们组装局部 Prompt，调用工具（如弹劾、投靠）后，再次休眠。

### 10.2 应对 Context 膨胀的三大神器
1. **图数据库（Neo4j）按需裁剪**：只查询 Agent 关系网内的局部子图，避免上帝视角。
2. **动态摘要（Memory Summarization）**：后台定期对历史事件进行“记忆浓缩”（如浓缩为“你近期遭到政治打压”）。
3. **宏观局势抽象化**：传递结构化的权力指数，而非几百人的名单。

### 10.3 存档与状态管理 (Save/Load)
采用**切片快照机制**：回合开始时固化一次全量世界状态（World State，包含知识图谱、动态摘要、浮点数值等），回合内只记录指令序列，从而将庞大状态打包为体积可控的本地 `.sav` 文件，确保 S/L 时 Agent 记忆不断层。

## 11. 里程碑规划与自动化测试

从“演示版”向“完全体”的开发规划：
1. **观天下**：地图渲染与行政区划交互。
2. **廷议与明诏**：多 Agent 辩论逻辑与自然语言解析器 (Policy Parser) 结合二次确认机制。
3. **流程穿透**：引入 `court_flow_engine.py` 计算执行阻力与【圣旨追踪日志】。
4. **成长破局**：国策树与科技树。
5. **观因果**：灾荒->流民->叛乱的完整动态演化。

**自动化平衡测试系统（Headless Auto-Play）**：
开发无头模式脚本，代替玩家下达诏书，在无 UI 状态下自动跑 10,000 局，记录崩盘时间点与死因。借此反向调优权重公式中的常数，确保普通玩家有 10%-15% 的破局概率，避免数值失控。
